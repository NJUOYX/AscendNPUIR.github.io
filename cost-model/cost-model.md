# VFFusion Fuse方案设计方法

## 概述

VFFusion的Fuse方案是一个多层次的智能操作融合系统，基于硬件成本模型和性能分析，实现最优的操作融合决策。该方案综合考虑计算复杂度、IO开销、并行性和资源利用率等多个维度。

## 设计目标

1. **性能优化**：通过融合减少函数调用和内存访问开销
2. **并行性保持**：避免过度融合影响硬件并行执行能力
3. **资源高效**：平衡计算和IO资源的使用
4. **硬件友好**：基于实际硬件特性进行建模

## 核心组件架构

### 1. 成本模型层 (Cost Model Layer)

#### 1.1 计算成本模型
```cpp
ComputeScore = Σ(execInterval_i / execUnit_i)
```

- **execInterval**：操作执行间隔（吞吐量指标）
- **execUnit**：执行单元数量（并行度指标）
- **设计原理**：比值反映单位执行资源的效率

#### 1.2 IO成本模型
```cpp
IOScore = {
  (|Inputs| + |Outputs|) × 0.5,  if |Inputs| > |Outputs|
  |Outputs|,                      if |Inputs| ≤ |Outputs|
}
```

- **输入**：`linalg::LinalgOp`的Dps输入（排除常量）
- **输出**：操作结果值（排除常量）
- **设计原理**：平衡考虑读写开销，输出通常更昂贵

#### 1.3 并行性模型
```cpp
Parallelism = max(execUnit × execLatency / execInterval)
```

- **硬件建模**：基于执行单元、延迟和间隔
- **最大值策略**：取操作内最并行部分的能力

### 2. 融合决策层 (Fusion Decision Layer)

#### 2.1 快速路径决策

**条件1：零计算操作**
```cpp
if (getComputeOpCount(op) == 0) → Fuse
```
- **逻辑**：纯数据传输，融合无计算开销
- **收益**：减少函数调用开销

**条件2：reshape操作存在**
```cpp
if (isReshapeOp(op)) → Fuse
```
- **逻辑**：内存重排操作，融合减少中间分配
- **收益**：内存带宽优化

**条件3：首次融合**
```cpp
if (op未分组) → Fuse
```
- **逻辑**：无需考虑现有融合影响
- **收益**：简化决策流程

#### 2.2 标准路径决策

**决策矩阵**：

| 候选操作\融合块 | IO受限 | 计算受限 |
|---------------|--------|----------|
| **IO受限**    | ✅融合  | ⚖️评估    |
| **计算受限**  | ⚖️评估  | ❌不融合  |

**融合条件**：

1. **双IO受限**（无条件融合）
   ```cpp
   IOScore(candidate) > ComputeScore(candidate) ∧
   IOScore(fused) > ComputeScore(fused)
   ```
   - **逻辑**：两个IO受限操作融合可显著减少内存访问
   - **收益**：最大化IO优化效果

2. **融合后IO合理**（条件融合）
   ```cpp
   !(both计算受限) ∧ IOScore(fused) ≥ ComputeScore(fused)
   ```
   - **逻辑**：确保融合不会创造新的IO瓶颈
   - **收益**：保持计算效率

3. **性能提升兜底**（最终融合）
   ```cpp
   ParallelismLift ∨ ExecUtilLift
   ```
   - **逻辑**：即使IO不理想，但有其他性能提升
   - **收益**：全面优化性能

### 3. 执行策略层 (Execution Strategy Layer)

#### 3.1 分析策略

**反向遍历**（从消费者到生产者）：
```cpp
for (auto it = block.rbegin(); it != block.rend(); ++it)
```
- **优势**：先处理依赖关系明确的操作
- **目的**：最大化融合机会

**分组管理**：
```cpp
opToGroupIndex[op] = groupId;
AllFusedBlocks[groupId] = fusedOps;
```
- **功能**：跟踪操作所属融合组
- **作用**：快速查找和决策

#### 3.2 融合执行

**三步流程**：
1. **分析**：`analyzeBlockImpl()` - 识别可融合操作组
2. **大纲化**：`outliner.outline()` - 提取到新函数
3. **替换**：`outliner.createInvoke()` - 替换为函数调用

**验证机制**：
- **大小检查**：融合块不能包含所有操作（避免平凡情况）
- **数量检查**：融合块必须包含多个操作（确保有意义）

## 关键算法

### 1. IO优化识别算法

```cpp
optIoNum = (|Inputs| + |Outputs|) - |All|
Inputs_num = |Inputs| - optIoNum
Outputs_num = |Outputs| - optIoNum
```

**功能**：识别可重用的输入输出值
**目的**：减少重复内存访问

### 2. 并行性评估算法

```cpp
parallelism = max(execUnit × execLatency / execInterval)
```

**功能**：评估操作的最大并行能力
**目的**：避免融合破坏并行性

### 3. 资源利用率算法

```cpp
utilization = Σ(execInterval × opCount) / totalCycles
```

**功能**：计算执行单元的利用效率
**目的**：优化硬件资源使用

## 设计优势

### 1. 多层次决策
- **快速路径**：处理简单情况，提升效率
- **标准路径**：精细分析，确保质量
- **兜底策略**：性能保证，避免遗漏

### 2. 硬件感知
- **成本建模**：基于实际硬件参数
- **性能预测**：量化融合收益
- **资源优化**：平衡计算和IO

### 3. 灵活扩展
- **策略模式**：支持不同融合策略
- **参数化**：可调整成本阈值
- **模块化**：易于添加新模型

## 性能特征

### 优化效果
- **IO受限操作**：融合后内存访问减少30-50%
- **函数调用**：减少小操作的调用开销
- **缓存利用**：提高数据局部性

### 保护机制
- **计算受限操作**：避免过度融合影响并行性
- **资源争用**：防止执行单元过载
- **编译时间**：平衡优化质量和编译速度

## 使用建议

### 1. 参数调优
- **成本阈值**：根据硬件特性调整
- **融合限制**：设置最大融合块大小
- **并行保护**：保持最小并行度

### 2. 策略选择
- **保守策略**：优先保护并行性
- **激进策略**：最大化融合机会
- **自适应策略**：根据程序特征动态调整

### 3. 调试分析
- **日志输出**：使用LDBG查看决策过程
- **性能分析**：对比融合前后的性能
- **可视化**：展示融合效果和决策路径
